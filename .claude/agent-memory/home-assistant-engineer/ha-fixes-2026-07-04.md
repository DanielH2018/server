**1. Bedtime prompt now has a TWO-PART suppression escape — complementary, NOT redundant.** Don't
"simplify" either away. (a) `sensor.bedroom_winddown_start` availability `> now()` (templates.yaml,
sensor-side) → catches a STALE PAST alarm (Wear OS holds a fired alarm) → sensor unavailable → 22:30
fallback fires. (b) `bedroom_bedtime_prompt`'s fallback gate got a third OR term
`as_datetime(states('sensor.bedroom_winddown_start')) < now()` → catches the DIFFERENT case where the
alarm is FUTURE but its winddown (alarm−8h) is already PAST (alarm set after alarm−8h, e.g. a 06:00
alarm created at 23:00 → winddown 22:00). Here the sensor stays available with a past value, so (a)
can't help and the dynamic `at:` has nothing future to fire on. The leading `in ['unknown',
'unavailable']` term short-circuits so `as_datetime` never runs on a non-timestamp.

**2. `bedroom_fallback_wake` rewritten STATELESS (real rewrite, not a tweak).** Was a `repeat.while`
loop that stayed in-process ~45 min → an HA restart/deploy mid-window killed it and the ramp did NOT
resume. Now `time_pattern: minutes: "/1"` mirroring `bedroom_wake_ramp`: each tick recomputes `elapsed`
from the persisted anchor (`input_datetime.bedroom_last_wake` + latched
`input_number.bedroom_wake_sleep_min_latched`) so a restart resumes on the next minute. Design points:
- The fallback has NO `bedroom_morning_reset` to do setup, so it does its OWN one-time setup in a
  START branch gated `no_real_wake and in_start_window([06:00,06:45)) and not wake_today`. Stamping
  last_wake flips `wake_today` true → START runs EXACTLY once; later ticks hit the RAMP branch.
- `no_real_wake` (ws unavailable OR ws.date != today) gates EVERY branch → never double-drives the
  lights with `bedroom_wake_ramp` (a today-dated wake_start, available even post-fire under the 2h
  grace, defers the fallback — consistent with the 07-02 grace analysis).
- Bounded RELEASE branch (`in_wake_release_window(elapsed)` [45,90) + `al_still_manual`) hands off to
  AL, exactly like wake_ramp's `release_due`.
- Good-morning note MOVED to START (was after the loop) → also fixes 07-02 finding #4 (greeted even
  after a manual cancel): person-home is a top-level condition, leaving mid-ramp stops the automation.
- All light actuation still via `script.bedroom_lights_set(reason: wake_fallback)` (sanctioned writer).

**3. templates.yaml winddown comment** rewritten to state the wake_start divergence explicitly:
wake_start uses `> now() - 2h` (grace to outlive its in-flight 45-min ramp+release WINDOW); winddown
is a one-shot anchor with no post-fire window so bare `> now()` is correct; and `> now()` on the alarm
does NOT cover the "winddown past, alarm future" case (handled automation-side per #1).

**4. `bedroom_color_track` wake-suppression** now covers the fallback: when `sensor.bedroom_wake_start`
is unavailable it falls back to elapsed since `input_datetime.bedroom_last_wake`, suppressing tracking
in [0,45). Both real + fallback ramps now suppress uniformly (was latent, masked by the ±150K band).
